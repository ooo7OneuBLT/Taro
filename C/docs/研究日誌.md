## 2026-07-10 Phase 0（続き）：詰まり2件と解消、起動成功

* **詰まりA**：`pip install -e MIMo/` が `ModuleNotFoundError: No module named 'pkg\_resources'` で失敗。
原因：MIMoの`setup.py`が`pkg\_resources`を直接import。pipのビルド分離が新しい`setuptools`（pkg\_resources非同梱）を隔離環境に取得してしまうため。

* **詰まりB（＝躓き①の正体、想定と一部違った）**：単に`return`が無いだけでなく、`super().\_initialize\_simulation()`の戻り値を**受け取ってすらいない**ため、直後の`self.dt`参照（`self.model`に依存）が`AttributeError`になっていた。
対処：`self.model, self.data = super().\_initialize\_simulation()`と明示的に代入するよう1行修正（`MIMo/mimoEnv/envs/mimo\_env.py` `\_initialize\_simulation()`内、コメント付きでローカルパッチとして明記）。

## Phase 4 手順②③：HybridEnvラッパーを実装・検証

`bridge/hybrid\_env.py`（`HybridEnv(gymnasium.Wrapper)`）を新規実装。MIMoを無改変のまま包み、太郎の内臓を同居させて1つの身体として動かす層。

**実装した中身**

* `advance\_body\_one\_second()`：内臓の毎秒処理を、Taro `core\_b.py` `tick\_body()` の順序どおり忠実に再現（胃.tick → 血糖.receive\_glucose(吸収量×効率3.0) → 血糖.tick → 内受容.update\_from\_body → 内受容.tick → 胃.grow）。肺は発声/聴覚サブシステムの部品なのでPhase 4では除外。
* **時間の刻み対策**：`step()`のたびにカウンタを+1し、`STEPS\_PER\_BODY\_SECOND`（=100）に達したら内臓を1秒進める。太郎側の定数は無改変。
* **観測の合成**：MIMoの観測dictに `interoception`（\[hunger, sleepiness, discomfort, arousal]）を追加。観測空間にもこのキーを登録。
* **報酬の合成**：`homeostasis.compute\_reward()`（つらさが下がったら正）をMIMoの報酬に加算。
* **簡易授乳（★スタンドイン）**：空腹>0.6で授乳中でなければ自動授乳。本物の親シミュレータは後で差し替え。
* **設計判断**：`insula`（内受容→脳への変換）は「脳側」の部品なのでラッパーでは動かさず、生の内受容ベクトルを観測に足すだけにした。insula変換は脳と繋ぐPhase 5で行う。

**検証（`bridge/test\_hybrid\_env.py`、脳の代わりにランダム行動）**

* テストA（統合）：MIMoを通して300ステップ完走。クラッシュなし。観測に`interoception`が入り、hungerが[0,1]に収まることを確認。
* テストB（身体ダイナミクス）：内臓は「1秒=100 MIMoステップ」と遅い（人間の空腹は数時間スケール）ため、MIMoを数百万ステップ回すのは非現実的。そこで`advance\_body\_one\_second()`を直接3万回まわして高速検証。結果：**hungerが0.0〜0.604で上下し、授乳が1800秒発生し、恒常性報酬が正になるイベントが459回**。空腹が約6〜7時間かけて上がり閾値0.6で授乳が入り0に戻る、という実際の乳児のリズムを再現できた。

**Phase 4 総括**

* 図の下半分（ラッパー＋MIMo＋太郎内臓）が完成。**MIMo（外の身体）＋太郎の内臓（内の身体）が、1つの身体として動作する**ことを実機で確認。
* 脳はまだ繋いでいない（ランダム行動で身体側だけを検証）。
* CNNの扱いについて設計判断を確定（詳細は開発計画.md）。既存AI研究として正直にラベルして進める方針。

`bridge/sensory_encoders.py`を新規実装。`insula.py`（内受容感覚の通訳、太郎から流用）と同じ考え方で、固有感覚・前庭覚・触覚の3つに専用の変換層を追加。

**実装した中身**

## Phase 5（後半）：視覚エンコーダ（CNN）を実装

`bridge/vision_encoder.py`を新規実装。ステレオ視覚（eye_left・eye_right、各256×256×3）を、他の感覚と同じ64次元に圧縮するCNN。左右は同じ重みのCNNで処理（＝同じ視覚野を両目で共有、という想定）し、統合して1本のベクトルにする。

**詰まりと解消**：MIMoの画像（OpenGL由来）は上下反転のビューとして負のstrideを持っており、`torch.as_tensor()`にそのまま渡すと`ValueError: negative strides are not currently supported`で失敗した。`numpy配列を.copy()`してメモリを実体化してから変換することで解消。

**検証**：ダミー画像で1回の変換に約49ms。パラメータ数は約213万。**この49msは今後の速度最適化の課題として残る**（Phase 2で測った204 steps/secという速度は視覚エンコーダのCNN計算を含んでいないため、実際に学習ループに組み込むと相応に遅くなる見込み）。

## Phase 6：感覚運動の脳（試作）を実装

**重要な設計判断**：太郎の本体の脳（`cortex.py`のTaroBrain）は音声・発話計画・語彙と密結合した複雑な設計になっており、直接拡張するのはリスクが大きいと判断。そこで、Phase 6/7の「配線が正しく動くか」の検証専用に、**別の・小さな試作の脳（`bridge/sensorimotor_brain.py`のSensorimotorBrain）を新規に作った。TaroBrain本体には一切触れていない**。本物のTaroBrainとの統合（音声・発話・満腹予期と同じ脳の中で身体運動も扱えるようにする）は、この試作で配線を検証した後の、より大きな後続作業と位置づける。

**設計**：GRU（TaroBrainと同じ発想の1層再帰ネット）が、5感覚を束ねた320次元ベクトルを受け取り、①次に来る感覚を予測する（感覚運動予測ループの中核）②MIMoの90関節への運動命令を出す、の2つを行う。

根拠ラベル：GRU自体はTaroBrainと同じ【既存AI研究】。「次の感覚を予測し外れたら修正する」原理は【人間模倣】（予測処理、DIVA）だが、具体的な実装（320次元ベクトルのMSE予測）は簡略化した近似で⚠️今後の検証・改善の余地がある試作段階。

## 方針転換：試作の脳ではなく、本物のTaroBrainを拡張する方式に切り替え

運動性喃語（motor babbling、生後すぐから見られる本物の発達現象。声の喃語より早く始まる）を最初の学習対象とすることに決定。その実装方法を再検討し、**独立した試作の脳（sensorimotor_brain.py）ではなく、本物の`cortex.py`（TaroBrain）をコピーし、サブクラスで運動性喃語専用の経路を追加する方式**に切り替えた。

**懸念していたリスク**：TaroBrainは音声・語彙と密結合しており、直接拡張するのは大工事になると予想していた。

**実際にやってみた結果（想定より軽く済んだ）**：
- `taro_brain.py`（`cortex.py`のコピー、import文1行のみ修正）と、その依存`vocal_tract.py`（外部依存ゼロ、無編集でコピー）を`bridge/src/brain/`に追加
- **既存のメソッドは一切変更せず**、`TaroBrainWithMotor`というサブクラスで新規追加：
  - `sensory_proj`：5感覚融合ベクトル(320次元)を、音声embeddingと同じ次元に変換する層
  - `sensorimotor_prediction_head`：次に来る感覚を予測する
  - `motor_head`：関節命令を出す
  - `step_motor()`：この3つを使って1ステップ処理する新メソッド（既存の`forward_hidden`等、音声関連メソッドには触れない）
- `body_state_dim=0`で構築することで、`critic.py`への依存も回避（内受容感覚は既に5感覚融合ベクトルに含まれるため二重に持つ必要がない）
- **単体テスト・実際のMIMo経由の結合テスト（50ステップ）とも成功**。想定していたほどの大工事にはならなかった

**この設計の利点**：`sensory_proj`の出力を、既存の`self.gru`（音声用と同じ重み）にそのまま渡しているため、**将来「音声と運動を同じ隠れ状態で統合する」ときの土台がすでにできている**。試作の脳（sensorimotor_brain.py、既に動作確認済みで削除はしない）は、シンプルな比較対象として残す。

**次にやること**：`TaroBrainWithMotor`の`motor_head`に、方策勾配（REINFORCE）で実際に学習させる。報酬は感覚運動予測誤差（予測が当たりやすい動きほど良い＝運動性喃語の再現）。学習を重ねて予測誤差が下がる傾向が出るかを確認する。

## Phase 8：運動学習に必要な部品は、ほぼ全て太郎に既存だった

方策勾配の学習ループを設計するにあたり、いくつかの部品が「太郎に既にあるのでは」と気づき、確認したところ、実際にほぼ全て既存だった：

- **報酬の考え方**（予測が当たると心地よい）：`instincts/prediction.py`の`compute_prediction_reward()`。既に`core_b.py`で`compute_total_reward()`経由、実際の学習に組み込まれ稼働中
- **探索の強さ**（NE / 青斑核）：`instincts/locus_coeruleus.py`。報酬が少ない時期は探索を強め、報酬が十分なら弱める。音声に固有の実装ではなく汎用
- **方策勾配そのものの計算**：`basal_ganglia.py`の`TaroLearner.learn_action()`。中身は「選んだ行動のlog確率」と「報酬予測誤差δ」だけを扱う完全に汎用的な数式で、トークンや語彙への参照が一切ない

一方、**喃語を実際に生成するコード（`generate()`）は音声に深く依存しており、運動には転用できない**（発声のどこで止めるかの判定・調音器官の駆動など、中身が全部言葉の都合でできているため）。

そのため実装方針：`dopamine.py`・`locus_coeruleus.py`・`basal_ganglia.py`を`bridge/src/brain/`に無編集でコピーし（太郎からの流用）、`TaroBrainWithMotor.step_motor()`に、運動を正規分布からサンプリングして対数確率を返す仕組みだけを新規追加した（`generate()`の仕組みは使っていない）。誤差→報酬の変換は、太郎の既存の型（`imitation.py`の`max(0, 1 - dist/max_len)`＝誤差が小さいほど1に近い報酬）を踏襲しつつ、感覚のMSE誤差には`max_len`のような自然な上限が無いため、恣意的な正規化定数を持ち込まない`1/(1+誤差)`を採用した。

## 実装のやり直し：感覚エンコーダの「崩壊」

`bridge/tests/test_phase8_motor_learning.py`を書いて300ステップ回したところ、以下の問題が順に見つかった。

**1回目**：感覚エンコーダ（触覚・視覚等の通訳層）をオプティマイザに含め忘れ、ランダムな重みのまま固定されていた（`TaroLearner`は`brain.parameters()`しか見ないため）。修正し、脳＋感覚エンコーダをまとめて学習対象にした。

**2回目（本質的な問題）**：感覚エンコーダも一緒に学習させたところ、予測誤差が数万→9600万へ発散した。診断の結果、**「次の感覚」という予測の正解自体が、今まさに学習中の感覚エンコーダの出力**であることが原因と判明。エンコーダの出力にLayerNorm（学習パラメータなし、平均0・分散1に正規化するだけ）をかけて発散は止まったが、その代わり予測誤差が完全に0まで下がるという、別の意味で不自然な結果になった。

**3回目（崩壊の実測）**：学習後、全く違う2つの身体状態（リセット直後／5歩ランダムに動いた後）を融合ベクトルに変換して比較したところ、320次元のベクトルがほぼ同じ値になっていた（差はベクトルの大きさの2%程度）。予測head側の出力も同様。**「感覚を正しく予測できるようになった」のではなく、「どんな入力が来ても、ほぼ同じ一定の値を出す」という近道を学習していた**。これは以前確認済みの「衛星ヘッド崩壊」（体の状態だけで正解できる近道が常に成立し、語が不要になった事例）と構造的に同じ問題。

**原因**：予測に使う側と、正解を作る側が、同じ・同時に更新される重みを共有しているため、両方が「入力を無視して定数を出す」方向に一緒に動いても、誤差が簡単にゼロになってしまう（自己教師あり学習で知られる collapse という落とし穴）。

## 対策の根拠：恒常的シナプススケーリング（Homeostatic Synaptic Scaling）

対策案を3つ検討し（1. 正解側だけ更新をゆっくり追従させる／2. 出力のばらつきが下がったら罰する／3. 感覚エンコーダを一旦凍結し脳側だけ先に学習）、それぞれの人間模倣としての妥当性をWeb検索で確認した。

結論：案3（凍結→解凍）は「何ステップで解凍するか」という新たな恣意的な定数が必要になる上、人間の感覚野が文字通り「完全に凍結」した状態を経る訳ではないため、根拠は薄い。

対して、**恒常的シナプススケーリング**（Turrigianoらの研究）という、脳に実在する仕組みが今回の問題に直接対応することが分かった：
- 「予測誤差から学ぶ」ヘブ型の仕組み（今回のbackprop）は、放っておくと不安定化し、活動が0に潰れるか暴走するかのどちらかに転びやすいことが理論・実験の両面で知られている
- これに対抗するのが恒常的シナプススケーリング：神経細胞は「これくらいの活動量が普通」という基準値（set point）を持ち、実際の活動がそこから外れるとシナプスの強さ全体を底上げ／底下げして基準値に戻す
- 発達期の大脳皮質感覚野で機能していることも確認されている

「予測誤差からの学習」と「暴走・崩壊を防ぐ恒常性」は脳の中でも別々の・並行して働く仕組みであるため、今回もこれを、NE・ドーパミンと同じ「本能」として実装する方針とした（設計のみ確定。実装はこれから）。

参考文献は`docs/参考文献リスト.md`にまとめた。

**自己訂正メモ**：当初「案2（ばらつきで罰する）は根拠のない工学的トリック」と即断したが、確認せずに判断したための誤りだった。実際には恒常的シナプススケーリングと機能的に同じであり、正しく実装すれば人間模倣として十分成立する。裏付けを取らずに「これは工学的」「これは人間模倣的」と判定しないこと（次節の教訓として記録）。

## 教訓：新しい仕組みを作る前に、太郎の既存コードを確認する

このセッションでは、NE（探索）・予測誤差報酬・方策勾配の計算という3つの部品について、一度は「MIMo用に新しく設計する」つもりで検討を始めたが、実際には全て太郎に既存で、そのまま流用できるものだった。指摘を受けて都度確認し直す形になった。

## 崩壊問題の根本原因を特定・解決（複数エージェントによる討論）

その後、GRU分離（音声用と運動用）・PV-RNNに着想を得た確率的潜在変数（`predictive_coding_latent.py`）・独立した固定の目標エンコーダ（`target_fusion`）と、いくつもの対策を積み重ねたが、8シードで検証すると3/8しか健全にならず、根本解決には至っていなかった。

そこで、ML工学・予測符号化の忠実性・力学系・一般ML文献・予測符号化/PV-RNN研究の文献という5つの視点から複数エージェントに調査させ、3ラウンドの討論をさせたところ、**3ラウンドとも一致して同じ原因**に到達した：`predictive_coding_latent.py`の`prior_net`（"事前の予想"を作る部品）が、コード中のすべての使用箇所で`.detach()`されており、**一度も学習されずランダム初期化のまま固定され続けていた**。これが、KLダイバージェンスの分母（`exp(logvar_p)`）を、学習で補正されないシード固有の値にしてしまい、「一部のシードだけ崩壊する」現象の直接の原因になっていた。あわせて、力学系の視点からは「方策勾配（REINFORCE）が`z`を経由して感覚エンコーダまで逆伝播し、高分散なノイズを知覚学習に混ぜている」ことも指摘された。

**対策**：①`prior_logvar`・`post_logvar`を安全域`[-4, 4]`にクランプ、②外側のKL損失で`prior_net`のdetachを外し実際に学習されるようにする、③`motor_head`に渡す`z`を`.detach()`し方策勾配が共有エンコーダへ漏れないようにする、の3点を実装。

**結果**：同じ8シードで再検証したところ、崩壊率が**5/8→1/8に改善**した。詳細は`docs/人間模倣からの逸脱リスト.md`のHに記録。一般ML文献調査からは「固定した目標エンコーダだけでは崩壊を完全には防げない（BYOLのアブレーションが示す通り、理論的にはシード依存の"引力域"問題が残りうる）」という知見も得られており、今回の1/8という残りは、これと整合する結果と考えられる。

## 測り方の見直しと、崩壊解決の最終確認

上記の「1/8」を出した崩壊チェックは「リセット直後 vs 5歩後」の1回勝負で、ランダムな5歩に依存して結果がブレることが分かった（実行するたびにシード5が崩壊したり健全になったりした）。今日1日で測り方のミスを4回・乱数のズレを3回やらかしたのを機に、`docs/検証の落とし穴チェックリスト.md`を作成し、崩壊チェックを「24個の多様な姿勢での融合ベクトルのばらつき（バッチ分散、未学習エンコーダ基準で正規化）」という安定した指標に置き換えた。

さらに、この新しい測定器そのものを検証した（チェックリスト項目6＝測定器の健康診断）：
- **Test A**：わざと定数を返す偽エンコーダにかけて 0.00% → 崩壊を正しく検出できる（測定器は健全）
- **Test B**：`prior_net`の修正をわざと元に戻した"壊れた版"を学習させると、シード5が0.3%まで崩壊（1/5）。**同じシード5が、修正版では81%（健全）**。バグ1個だけを変えた対照実験で結果が崩壊↔健全と反転し、検証済みの測定器がそれを正しく判定した。

**最終結論**：
1. 崩壊は**本物の現象**だった（測定の誤検知ではない。Test A・Bで実証）。ただし旧チェックは深刻さを**水増し**していた（直したシード5を「崩壊」と誤判定していた）。「崩壊は全部測定ミスだったのでは」という一時の疑いも、行き過ぎだった。
2. `prior_net`・`z.detach`の修正は、その本物の崩壊を**実際に解決していた**。信頼できる新チェックで、修正版は**13シード全て健全（崩壊 0/13）**。
3. motor collapse（運動のフリーズ）も別途検証し、起きていないことを確認（逸脱リストB1）。

課題1（感覚エンコーダの崩壊）・課題2（運動のフリーズ）はいずれも解決とみなせる。

## 評価指標の見直しと、自己モデルの"形成"の検証（2026-07-11）

崩壊解決の後、「順モデル（自己内部モデル）が確立したか」の検証（目標C1）に進んだ。当初は**「持続予測（＝"何もしない"と言い張るだけの予測）に、絶対誤差で勝てるか」を合格基準**にしていた。多角的に diagnose した結果、次々と formulation の問題が見つかり、各々を直すたびに改善した：①順モデルが**行動を入力に取っていなかった**（遠心性コピーを追加）②予測を32次元の潜在から"まるごと"当てていた（**残差予測**＝今の感覚＋変化量に変更）③予測の"正解"が崩壊防止の凍結ランダムエンコーダで難しすぎた（**固有感覚という自然な信号**に変更）。基準値比較は 191%→106% と一直線に改善したが、**それでも持続予測に勝てず、経験を増やしても互角の手前で頭打ち**になった。

**スケール不一致の発見と高速化**：この頭打ちを掘る中で、根本的な問題に気づいた——300ステップ＝シミュレート内わずか3秒で、人間が数ヶ月かけて作る内部モデルの"確立"を測っていた。対策として、この段階に不要な**視覚と触覚センサをオフにし、脳の判断を1秒に1回に間引く（frame-skip）**ことで、実行速度を **sim/real 0.18→5.83（約32倍速）** に改善（重かったのは触覚センサ計算74%と視覚レンダリングで、物理は激安と実測で判明）。これで初めて"数時間ぶんの人生"を現実的な時間で与えられるようになった。

**評価指標そのものの誤り**：ここで、そもそも「持続予測に勝つ」という基準が、時系列予測の一般的なサニティチェックであって、**自己モデルの分野の標準ではなかった**ことに気づいた。先行研究（発達ロボティクスの自己他者認知、乳児の随伴性検出）を当たると、内部モデル形成は**絶対精度ではなく「自己（随伴）vs 他者（非随伴）の予測誤差の対比＝分類」で機能的に判定**していた。汎用MLの直感で baseline を選び、ドメインの文献で裏取りしていなかったのが反省点（検証の落とし穴チェックリストに項目7として追加）。

**正しい指標で見ると結論が変わった**：自己vs他者の対比（入替テスト）と、予測変化と実変化の相関を主指標に据え直すと、太郎の自己モデルは **経験とともに強く伸びていた**（入替マージン +8.6%→+22%→+43%、相関 0.2→0.4）。この指標が本物であることは健康診断で確認（未学習の太郎では≈0）。一方の絶対精度の頭打ちは、**1秒先の全身関節が転倒・接触で予測不能な部分が大半という課題の性質**によるもので、モデルの欠陥ではないと結論した。

**PV-RNN化の是非**：予測が弱い一因として「推論時の誤差回帰が簡略（今の一瞬にしか合わせない）だから」を疑い、アブレーション（回帰の回数 0/3/10）で確かめたところ、**回数を増やしても予測は改善しなかった**。よって「回帰が浅いから作り替える」根拠は弱いと判断（多時刻の"窓"への拡張は別軸で未検証）。作り替える前に安いアブレーションで確かめ、無駄な改修を回避した。

**残る論点**：自己モデル形成の究極の判定は、先行研究では「自分用のモデルを他者に当てはめて他者を予測できるか」＝自己→他者の転用（目標D）。ただし他者の観測には視覚が必要で、真のDは視覚＋対応問題という大きな一歩。詳細な数値と判定は`docs/人間模倣からの逸脱リスト.md`セクションIに記録。

## 目標C2：睡眠リプレイによる自己モデルの定着（2026-07-13）

C1で「絶対精度の頭打ち」を"課題の性質"と結論していたが、真因をさらに掘った。**消去法**：学習率を0.001/0.005/0.01で振っても飽和点（持続予測比~80/変化相関~0.45）は動かず＝lrは無罪（3seed）。探索ノイズも、報酬ベースのNE調整が最初の10分で下限まで下げていて無関係と判明（「永久に高ノイズ」という当初の見立ては誤りで、実測のnoise列で訂正）。予測ホライズン(K)を短くする診断は、Kを変えると指標の意味・生きた時間まで同時に動く交絡で判定不能だった。

決め手は**公平対決**（予測構造だけを同じ土俵＝1秒先で比較）：素のMLP順モデルをオフライン多周回で学習すると相関0.58まで届いた（naiveな「0.1秒×10連鎖」ロールアウトは誤差の雪だるまで大負け）。＝**頭打ちはホライズンでなく、太郎の"オンライン1回きり学習（復習不足）"が原因**、と推理が確定した。

これは人間の睡眠中の記憶固定の欠落そのもの：既に目標Bには海馬（`hippocampus.py`）による睡眠リプレイがあるのに、Cの自己モデル学習はそれを使っていなかった。そこで**睡眠中に貯めた経験を再生して定着させるリプレイ**を実装し検証したところ、相関 0.44→**0.56**・持続予測比 82→**69**と頭打ちを突破（オフライン上限にほぼ一致）。自己/他者の区別（agency）も 83→**95%** に改善。独立した2手法（オフライン0.58・リプレイ0.56）の一致が、真因の確定を裏づけた。

**本実装（太郎C2-1）**：この機構を実験scriptから脳本体へ格上げした。運動用の海馬 `MotorHippocampus`（言語用海馬の"型"を感覚運動テンソル用に書き直したもの＝コピーではない）と、`TaroBrainWithMotor` に `infer_latent`（行動をGRU入力へ）・`predict_proprio`（残差予測）・`consolidate`（睡眠リプレイ）を追加。脳の公式メソッドだけでリプレイ効果を再現（改善幅 persist −12・corr +0.13、実験と一致）。＝太郎は**言語も自己モデルも同じ海馬リプレイで睡眠中に定着する**形に統一された。

## 逆モデル診断と予測幅スイープ（2026-07-14）

**予測幅スイープ：1秒(K=100)が最良、"短縮＝人間条件"仮説は棄却**：
- 動機＝「人間は短い予測幅+閉ループ→太郎も予測幅を短くすれば精度↑？」。指標はpersistでなく**行動依存(margin)/逆到達(star/random)**（persistは短い幅で持続予測が満点になり不公平）。

## Goal Babbling：目標指向で"逆モデル側"が伸びる（2026-07-14）

運動性喃語（発達段階）の中で、動かし方を **Motor Babbling（ランダム探索）→ Goal Babbling（目標指向）** へ半分移行（`C_GOALBABBLE`、`i%2`で50%）。目標＝経験バッファから過去の固有感覚をサンプル（Self-Priorの最小版）、逆モデル推論（凍結順モデルを勾配で反転）でそこに届く行動 a\* を逆算して据える。K=100・2シード・逆テスト込み。

**結果（Motor→Goal）**：
- **逆側が明確に改善**（2シード一致）：現実の目標到達 **star/random 0.55→0.45**、行動の復元 **recover_corr 0.32→0.50**。margin +47→+53、agency もやや上昇。
- **順モデルの精度は横ばい**：corr 0.534→0.535、err_areal ≈1.07で不変。

**解釈**：Goal Babbling は順モデルの予測精度でなく、**逆側（望む感覚→行動・目標到達）を直接鍛える**＝発達ロボティクスの定説（goal babbling は inverse model を効率よく bootstrap する, Rolf et al.）通りの効き方。＝**目標指向という機構自体が効くと実証**。

**暫定・次**：探索/目標指向の切り替えが `i%2` 固定＝非人間的（`doc/人間模倣からの逸脱リスト.md`に記載）。次は**NE（好奇心）・恒常性のズレから切り替え確率を創発**させる精緻化へ。ログ＝`logs/C/goalbabble/`。

### 切り替えの創発化（`C_GB_SWITCH`、2026-07-14）

`i%2`固定を、内的信号から創発する切り替えに置換。3方式を比較（star/random, 低い＝良）：
- **fixed（i%2, 50%）**：0.445（基準）
- **ne（NE単独）**：0.52。NEはCで10分で下限に張り付き幅がない→実質90%目標ねらいに退化→**やりすぎで悪化**（探索が減る）。＝**50%のバランスは偶然でなく効いていた**。
- **pe（予測誤差＋NEの足し算, 創発, 既定）**：**0.46**。motor(0.545)・ne(0.52)より良く、**fixed(0.445)にほぼ並ぶ**。＝**切り替えを驚き駆動の創発に置換しても効果は維持**、より人間的。ただし僅差でfixedに及ばず・seedばらつき大。

**位置づけ**：向き（驚き大・NE大→探索）は根拠あり（EFE/LC-NE/予測符号化）だが、**足し算・重み・正規化の式は恣意的**（逸脱ラベル済み）。完全な根拠づけ＝EFE直接計算（やることリスト）。＝**恣意的な50%を、根拠ある向きの創発に置換（効果同等・人間度↑）**を達成。

## C4：閉ループ制御 — MPC流用は失敗、予測幅を揃える（2026-07-14）

閉ループ vs 開ループの到達診断（`closed_loop_probe`, `C_CLPROBE`）を実装。目標姿勢g（過去に経験した姿勢）へ、開ループ（逆算した1行動を保持）/ 閉ループ（内側の刻みごとに実際の状態から再逆算して補正）/ random / nothing で到達距離を比較（MuJoCo save/restore, d_floor≈0で妥当）。

**第1版（補正の刻み＝K//8＝0.12秒、順モデルは1秒先予測）＝失敗**：closed/open ＝ 1.13, 1.16（2seed）＝**閉ループが開ループに負け**。closed/nothing≈0.95（目標へほぼ近づけない）。
- **原因＝予測幅のミスマッチ**：順モデルは1秒先を予測するのに、閉ループは0.12秒ごとに補正→ちょっとしか動かず、再計算しても状態がほぼ同じ→ほぼ同じ行動を出し直すだけ＋ジッター＋GIGO。＝**「1秒モデルをMPCで0.12秒補正に流用」は噛み合わない**（＝K=100で回してた点をユーザーが指摘）。

**修正方針（詰める）**：補正の刻みを**順モデルの予測幅に合わせる**（`k_inner = K`）。＝再計算のたびに、モデルが得意な予測幅で補正する。これが噛み合えば「閉ループ（横並びに補正）＞開ループ（撃ちっぱなし）」が出るはず。→ 再検証。
（さらに本命は、短K＋閉ループ込みの訓練で"短い先を予測するモデル"を作ること＝人間版。まずは既存モデルで予測幅を揃えて検証。）

**予測幅マッチ版・根拠づけ版（K=100）＝いずれも負け**：k_inner=K（1秒補正）に揃えても closed/open＝1.05〜1.09（reach_len=4固定）／1.17〜1.31（停止条件・reach_len≈2.2）。＝**K=100の閉ループは版に依らず開ループに負け**（補正が1秒間隔で粗く、2回で止まる浅いループ）。停止条件は「1回でも近づかなければ止める」でやや早すぎ（ノイズに弱い、要改良）。＝**K=100は閉ループに不適**と確定。

**短K(K=10)＋閉ループreaching訓練（held goal＋停止条件）＝失敗（せいぜい微増）**：閉ループ訓練 vs 開ループ(motor babbling) を短K同士で比較。margin +29 vs +25、corr 0.38 vs 0.37＝**目的的な動きは僅かに信号を足すが救うほどでない**。closed/open(probe)＝1.08〜1.11で**閉ループ制御も開ループに負け**。しかも短K全体が **K=100最良(margin48/corr0.53)に遠く及ばない**（margin29/corr0.38）。

**C4の結論（決着）**：**閉ループ単体では短Kを救えない**。真の壁は**"0.1秒で学習"すること自体**（信号がノイズに埋もれてモデルが育たない）。人間は「一生かけて（長いスケールで）学び、0.1秒では使うだけ＋時間積分＋多感覚融合」でこれを回避してる。＝閉ループは"使う側"の工夫で、"学習側"の工夫（①階層的時間スケール＝長い窓で学び短い間隔で使う ②時間積分 ③多感覚）が要る。候補は `doc/やることリスト.md`「短K学習の信号不足を効かせる候補」に記録。→ **C4＝閉ループ制御を探究し「単体では不十分、学習側の工夫が必要」と結論**。次の本命はこの候補群、または聴覚D。

## 次の一手：触覚が学べない原因＝「温度や質感が無いから」仮説（2026-07-15）

**ユーザーの仮説**：「人間は質感や温度で、これが人なのか物なのかを判断している」。

**裏づけ（実在の神経機構）**：[Ackerley et al. (2014) J Neurosci](https://www.jneurosci.org/content/34/8/2879) が微小神経電図法で人の神経を直接記録。前腕を 18℃/32℃/42℃ で撫でると、**C触覚線維(CT afferents)は人肌の32℃でのみ最もよく発火**し、**主観的な快さと発火頻度が相関するのも32℃のときだけ**。結論＝CTは「**人肌の温度で、ゆっくり撫でられる**」に特化して調律された、**人と人の肌の接触を知らせる末梢機構**。
分布：CTは毛のある皮膚（前腕・背中・顔）に多く、手のひらは**約7分の1の密度**（[Watkins et al. 2021, J Neurophysiol](https://journals.physiology.org/doi/full/10.1152/jn.00587.2020)。「手のひらに無い」という旧定説は更新された）。

**設計への含意＝人間の触覚は密度勾配が正反対の2系統**：

| | 密度が高い場所 | 用途 |
|---|---|---|
| 識別的触覚(Aβ) | **指先**が密（MIMo実測 指先0.002 vs 下腿0.038＝19倍） | 物の形・質感 |
| 情動的触覚(CT) | **前腕・体幹**が密（手のひらの7倍） | **人**（人肌の温度＋速さ） |

**MIMoが持っているのは前者だけ**＝完全に「物を触るための手」の設計。**人を感じる系統が太郎には1本も無い。** `infant_touch_params()` は部位ごとの密度を係数で操作できるので、**逆勾配のチャンネルを足す**のは構造的に無理がない。

## 触覚が学べない原因を特定：情報不足ではなく「予測対象の形」（2026-07-15）

**問い**：触覚 persist 97〜104%（学べない）の原因は、①学ぶ対象が無い ②情報が足りない ③学習側、のどれか。打ち手が正反対になるので切り分ける。

**ユーザーの仮説**：「人間は質感や温度で、これが人なのか物なのかを判断している」＝今の触覚は「どこかに何かが当たった力」しか伝えず、**何に当たったのかを区別する情報が無い**から予測不能。

**やり方**（`d_touch_predictability.py`）：太郎の実際の課題は「今の状態と行動から**0.5秒後**の触覚を当てる」。そこで**もっと簡単な問題**を出す＝「**今の**関節角度から**今の**触覚を当てる」。関節角度が決まれば体の形は幾何学的に決まるので、原理的には解けるはず。入力を3段階（①固有感覚621 ②＋前庭 ③真の物理状態qpos+qvel＝オラクル）にして、情報がどこで失われるかを見る。

**★測定器を3回作り直した（毎回「ユーザーの仮説を支持する」という答えが出たが、全部偽物だった）**：

| 版 | 落ちた健康診断 | 実際に起きていたこと |
|---|---|---|
| 1版 | 全部負・**オラクルが最悪(-29%)** | 300stepしか学習させておらず単なる収束不足。**情報を増やしたのに成績が下がる**のは収束不足のサイン |
| 2版 | **負例（入力=乱数）が学習R²+90.2%** | 3606次元の出力を3200標本で当てさせ、ネットが入力を無視して**丸暗記**。学習R²は全部ただの暗記だった |
| 3版 | **正例（入力=触覚そのもの＝恒等写像）が検証R²+60.3%** | 時系列で後ろ20%を切ったため、太郎が転がって別姿勢になった区間を検証にしていた＝外挿を要求 |

**最後に効いた対照＝「太郎が学べると分かっているもの（固有感覚）を、同じ測定器に通す」**。従来の正例（恒等写像）は「学習手続きが動くか」しか見ていない。**太郎の課題における正例**が要る。→ 落とし穴チェック項13。

| **触覚** | +77.9% | **−65.6%** |

**差93ポイント。** データ量・容量・分割の厳しさは**固有感覚側にも等しく効いている**のに、固有感覚は正・触覚は負＝**触覚という対象に固有の困難**。（参考：①固有感覚のみ→触覚 −34.1% ②＋前庭 −29.3%。健康診断＝正例(恒等)+96.7% / 負例(乱数)−15.9%で通過。）

**★判定の誤りを修正**：「オラクルが解けない＝情報が無い」と判定してはいけない。**物理が情報の存在を保証している**——関節角度が決まれば体の形は幾何学的に確定し、MuJoCoは決定論的なので (qpos, qvel) から接触力は数学的に一意に決まる。オラクルのR²が負でも意味するのは**「決まってはいるが実質的に学習不能なほど暴れる関数」**。スクリプトの自動判定にこの誤りを書いていたので修正した。

**結論＝ユーザーの仮説は"理由が変わって"生き残る**：
- ❌「情報が足りないから学べない」→ **物理的に成立しない**（情報は100%存在する）
- ⭕「**力だけでは足りない**」→ **実証された**。理由は「接触力が不連続でカオス的」だから。腕が1ミリ動けば接触がON/OFFで反転し力が跳ぶ。**部位ごとに畳んで55次元まで抽象化してもダメ**だった。

**打ち手＝「情報を足す」ではなく「予測対象を、暴れる力から滑らかな量に変える」**。力が学習不能な理由（不連続・カオス）は**温度には当てはまらない**：触れている相手が自分の腕なら、腕が1ミリ動こうが32℃のまま。＝ユーザーの言う「温度」の正体は**予測可能な形をした触覚**だった。

**次の検証**：温度チャンネルを実装し、**同じ測定器・同じ対比**にかける。温度の検証R²が固有感覚(+27%)側に来れば仮説は実証、触覚(−66%)側なら死ぬ。

## 温度チャンネルを実装し、同じ測定器で**棄却**した（2026-07-15）

**実装** `D/scripts/d_thermal.py`（`ThermalWrapper`＝観測に部位ごとの温度55次元を足す。物理には干渉しない）。設計の根拠：
- **人肌32℃・室温22℃**＝Ackerley et al. (2014) のCT線維が最もよく発火した「典型的な皮膚温(neutral)」そのもの。18℃/42℃では発火が落ちる。**恣意的な定数ではなく実験で測られた値**。
- **熱慣性（時定数1.0秒）**＝皮膚には熱容量があり温度は跳ねずに追従する。「力が学習不能な理由（不連続・カオス）」を構造的に持たせないのが狙い。※1.0秒は**根拠の弱い仮置き＝逸脱**。
- **部位ごと55次元**＝CTは密度が低く前腕・体幹に多い（手のひらは約1/7）。per-pointの3606次元は人間から遠い。

**単体テスト（何もしない太郎・仰向け）**：21/55の部位が人肌(1.000)を検出。**温度の変動係数7%（接触力は376%）**＝狙いどおり滑らかになった。

**★結果＝棄却**。同じ入力(真の状態225)・同じデータ・同じ分割・同じ出力次元(55)、変えたのは予測対象だけ：

| 予測対象 | 検証R² |
|---|---|
| 固有感覚（学べると分かっている） | **+26.5%** |
| **温度（今回追加）** | **−53.1%** |
| 接触力（学べない） | **−54.4%** |

**温度は接触力と1.3ポイントしか違わない。滑らかにしても、まったく学べるようにならなかった。**

**なぜ私の推論が外れたか（論点のすり替え）**：「腕が1ミリ動いても触れている相手が自分の腕なら32℃のまま」＝**時間方向**の滑らかさ。しかし予測に必要なのは**姿勢から温度を当てる**こと。温度の値を決めるのは「どこに触れているか」＝**接触そのもの**なので、**温度も姿勢の不連続な関数**になる。**時間について滑らかなことと、姿勢について滑らかなことは別物**だった。温度は接触の読み出し方を変えただけで、**接触の予測困難さをそのまま相続していた**。

**さらに裏側の問題**：熱慣性を入れた＝**0.5秒後の温度は「今の温度」でほぼ当たる**＝persistで勝てない。**滑らかにするほど「何もしない予測」が強くなり、行動依存(margin)が消える。** 温度は両方向から詰んでいた。

**生き残る事実**：接触に由来する信号は、姿勢からの予測が原理的には可能でも実質的に不可能。固有感覚は可能。＝**触覚を予測対象にすること自体が誤り**の可能性。

**次**：`d_train_self.py`（今朝「放棄」と印を付けたスクリプト）にあった設計＝「**触覚は状態zへの入力（文脈）として持つが、予測対象は自分の固有感覚**」。当時は別の理由（制御可能性比という誤った代理指標）で捨てたが、**設計自体は今のデータと整合する**。→ `C_TOUCH_MODE=input` を実装して検証する。

## ★バグ：仰向けのランはシードで再現できていなかった（2026-07-15）

### 決定実験：触覚は「力の回帰」でなく「接触の分類」なら学べるか（`d_touch_classify.py`）

ユーザーの判断で deep-research は打ち切り、**自前の測定に戻った**。`d_touch_predictability.collect()` をそのまま流用（同じ入力・同じデータ・同じエピソード単位の分割）し、**変えるのは予測対象の形式だけ**：力の合計（連続値）の回帰 → 部位ごとの接触の有無（二値）の分類。指標は**AUC**（0.5=でたらめ、1.0=完璧）。

**なぜAUCか**：正答率は使えない。寝ているだけで1202センサ点中589点(49%)が反応している＝「常に触れている」と答えるだけで高得点が出る＝**暗い部屋問題の分類版**。さらに**接触率10〜90%の部位だけ**を採点（初版は「少数派3標本以上」だけで通していたが、800中797回触れている部位も通ってしまう＝**落とし穴11の希釈が分類版で再発**しかけた。実測では10-90%が43-45部位あり、フィルタ前後で0.622→0.632とほぼ動かなかったので、この件では希釈は起きていなかった）。

**★結果（すべて検証AUC・変動する43部位の平均）**：

| 入力 | 次元 | AUC |
|---|---|---|
| 正例：触覚→接触（恒等） | 55 | **0.964** ✅測定器は健全 |

**★★副産物の発見：オラクルが固有感覚に負けた＝トルクが接触の近道**

①②（0.68-0.69）が③オラクル（0.632）を**上回った**。接触は (qpos,qvel) から決定論的に決まるので、その下流である固有感覚が上回るのは**論理的におかしい**。唯一ありうる説明＝**固有感覚が「答え」を直接持っている**。切り分けた：

固有感覚621次元の内訳（実測。連結順は辞書キーのアルファベット順）：
`actuation[0:180]=180 / limits[180:273]=93 / qpos[273:366]=93 / qvel[366:459]=93 / torques[459:621]=162`

| 入力 | 次元 | AUC |
|---|---|---|
| ①固有感覚 **全部** | 621 | **0.690** |
| ④固有感覚 **−トルク** | 459 | **0.652**（−0.038） |

**＝トルクのみ(162次元)が、物理の真の状態(225次元)を上回る**。そして**関節角度のみ(0.628) ≈ オラクル(0.632)**＝どちらも「純粋な幾何」で同じ水準。

**解釈**：`DEFAULT_PROPRIOCEPTION_PARAMS` は `["velocity", "torque", "limits", "actuation"]`＝**固有感覚は torque（関節にかかる力）と actuation（筋の活性）を含む**。これは「幾何から接触力を計算する」という難しい関数を**迂回**して、接触の答えを直接読み出している。人間で言えば**腱器官（Golgi tendon organ）・筋紡錘**にあたり、**生物学的にも正しい**。

**★この発見の含意（仮説）**：18本の実験で確立した「**触覚を入力に足しても自己モデルは良くならない（input ≈ off）**」は、これで説明がつくかもしれない——**太郎の固有感覚は既に接触を感じている**ので、触覚は**冗長**。＝「触覚は無害だが無益」は失敗ではなく**当然の帰結**だった可能性。ただしトルクの0.662も弱いので「完全に冗長」とは言えない。**要検証の仮説**。

### 視覚導入を実測評価＝設計書の懸念（メモリ・速度）は当初想定より遥かに軽い（2026-07-16）

ユーザー判断「触覚だけは厳しい／視覚を検討／独自性は視覚なしだけじゃない／とりあえずやる」。

**★最大の発見：視覚ONでもテクスチャを落とせる＝メモリはほぼ増えない**。`mimo_lean.strip_textures` は従来 `vision_params is None` のときだけ適用していたが、**この構成では養育者が赤カプセル**なので、落とすテクスチャ（顔の表情7種＋服の柄＝976MB）は**他者識別に一切無関係**。視覚ONのまま落とせる：
| 構成 | メモリ/本 | 速度 | 並列本数の目安 |
|---|---|---|---|
| 視覚OFF（現状） | 153MB | 1.7ms/step | ~22本 |
| 視覚ON・テクスチャそのまま | 2797MB | 3.4ms/step | ~6本 |
| **視覚ON・テクスチャ落とし(64x64)** | **232MB** | 3.4ms/step | **~15本** |

＝設計書が視覚を保留した最大の理由（1本2.8GBに戻る）は**この構成では消える**。速度は2倍増だがGPUで吸収。太郎の目に赤い画素105個を検出＝養育者は視覚obsに写る（ただし目線カメラか外部カメラかは要確認・下記）。

**視覚が他者理解に効く構造的理由（触覚との違い）**：
| | 触覚 | 視覚 |
|---|---|---|
| 他者の信号量 | 3.9%（床82.5%に埋もれる） | 常に画面に写る |
| 接触が必要か | 要る（触れないと相手が消える） | 不要（離れていても見える＝先読みの余地） |
| 予測対象の性質 | 不連続・カオス（0.71が天井） | 画素は相手が動けば滑らかに変化 |

**独自性について**（ユーザー判断）：「視覚なし」は独自性の一部でしかない＝視覚ONでよい、と決定。＝設計書の「触覚のみで試し→ダメなら視覚追加」の分岐に、実験結果（触覚は0.71の天井）を根拠に到達した。

### 視覚を「とりあえず動かして」判明した2つの重要事実（2026-07-16）

視覚版環境 `d1_carer_vision_env.py`（`CarerEnv`継承・視覚ONでもテクスチャ落とし・養育者は赤`CARER_RGBA`）を作り、**太郎の目に実際に何が写るか**を確認したところ、設計を左右する2つが判明した。

**★事実1：仰向けの太郎の目は「空」を見ている＝胸に触れる手は視覚に入らない**
- `obs['eye_left']`（gymのvision経路）は**外部視点の映像**を返していた（太郎の全身が外から見える＝眼球視点ではない）。標準の `MIMoBenchV2-v0` でも同じ＝**私の環境のバグではなく、gymのrender経路の問題**（MuJoCoの生API `mujoco.Renderer`＋`mjCAMERA_FIXED` で `eye_left` を直接描画すると正しい一人称視点になる）。**＝太郎の脳に視覚を渡す前に、この経路を必ず直す必要がある**（今のままだと脳は「外から自分を見た映像」で学習してしまう）。
- 生APIで得た**正しい眼球視点＝ほぼ空だけ**（赤画素0/16384）。仰向けだと目は真上を向き、胸(z=0.06)に触れる手(z=0.18)は視界の外。**天井を見上げた赤ちゃんが自分の胸を触られても見えない**のと同じ＝発達的に正しい。

**★事実2：手を「顔の上」に持ってくると、はっきり写る（発達的に正しい構図）**
生API眼球視点での赤画素（16384中）：
| 手の位置 | 赤画素 |
|---|---|
| 顔の上 z=0.30（近い） | **3911（24%）** |
| 顔の上 z=0.35 | 1794 |
| 顔やや足側 z=0.35 | 1581 |
| 顔の上 z=0.45（遠い） | 635 |
＝**覆いかぶさってくる養育者を、仰向けの太郎は見上げる**。カンガルーケア・あやし・face-to-faceの相互作用そのもの。

**★含意：触覚と視覚は"別の場所"を捉える＝相補的**
- 触覚：胸への接触を捉える（が床82.5%に埋もれる・目には見えない）
- 視覚：顔の上の養育者を捉える（が胸の接触は見えない）
＝**顔の近くで養育者が触れれば、見える＋感じるが同時に立つ**。これはAckerleyのCT触覚線維の「顔をゆっくり撫でる」条件とも、発達的なface-to-faceとも一致する。＝次は**養育者を胸でなく顔の周りに配置**するのが筋。

**メモリ/速度の実測（視覚ON・テクスチャ落とし）**：64x64で232MB・3.4ms/step（視覚OFFは153MB・1.7ms）＝**並列15本は保てる**。設計書の懸念は解消。
