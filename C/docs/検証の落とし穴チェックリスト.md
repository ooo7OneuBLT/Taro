# 検証の落とし穴チェックリスト

作成：2026-07-10
目的：実験・測定で「壁に当たった」とき、または測定結果を信じる前に確認するリスト。
2026-07-10の1日だけで、測り方のミスを4回・乱数のズレを3回やらかした反省から作成。
**測定結果を根拠に何かを結論づける前に、必ずこのリストを一通り確認する。**

---

## 1. 交絡（測りたいもの以外が混ざっていないか）

測っている数字に、目的以外の原因が紛れ込んでいないか。

- [ ] その数字は、本当に「知りたいこと」だけを反映しているか？
- [ ] 別の原因（重力・ノイズ・外部からの操作・初期化の偏り等）が混ざっていないか？
- 実例：
  - **描画速度の誤測定**：計測中にキー（一時停止）を押していて、実時間だけが進み速度が1/7に見えた
  - **motor collapseの誤判定**：「関節が動いている」を根拠にしたが、その動きは重力（体が崩れ落ちる）＋探索ノイズ（常に加わる揺らぎ）でも生じる。脳が学んだ運動とは限らなかった → ノイズ・重力を排除した「素の運動司令が入力に反応するか」で測り直すべきだった

## 2. サンプル数（1回勝負になっていないか）

- [ ] たった1回・1組の比較で判断していないか？
- [ ] 複数回・バッチ（多数の例）で平均を取ったか？
- 実例：**崩壊チェック**を「リセット直後 vs 5歩後」の1組だけで測り、ランダムな5歩に依存して結果がブレた。24個の多様な姿勢でのバラつき（バッチ分散）に変えたら安定した

## 3. 再現性と乱数（シードを固定したか・乱数をズラしていないか）

- [ ] 乱数シードを固定して、同じ結果が再現されることを確認したか？
- [ ] **測定用のコードが、学習より前に乱数を消費して、学習の乱数の流れをズラしていないか？**（測定は学習の後に置く。学習前に乱数を使うなら別の乱数系統にする）
- 実例：GRUを1個増やす／学習前にプローブ用の観測を集める、などで乱数がズレ、「同じシード」のはずが別の結果になった（3回）

## 4. 基準値（良い/悪いを判断する物差しがあるか）

- [ ] 「この数字は大きい/小さい」を判断する基準があるか？
- [ ] 未学習・ランダム初期状態・既知の正常/異常なケースと比較したか？
- 実例：崩壊チェックで「学習していない同じエンコーダのバラつき」を基準に併記することで、初めて「学習後がどれだけ保たれているか」を%で言えるようになった

## 5. 数式の事前検算（0コスト診断を先に）

- [ ] 実験を回す前に、紙の上で「この式は意図した効果を生むか」を確認したか？
- [ ] 高コストな実験の前に、低コストで分かることを先に潰したか？（手順.mdの「0コスト診断優先」）
- 実例：**precision式のミス**。`predicted.detach()`を混ぜる式が、展開すると`(1-precision)²`倍のただのスケーリングに退化していた。15回×2セットの実験を回す前に、数式を展開していれば1分で気づけた

## 6. 測定器そのものの健康診断（メタ検証）

- [ ] その測定方法が「本物の異常」をちゃんと異常と判定できるか、既知のケースで確かめたか？
- [ ] わざと壊したケース・明らかに正常なケースを入れて、測定器が正しく反応するか確認したか？
- 実例：新しい崩壊チェックを信じる前に、「わざとバグを戻した太郎」にかけて、ちゃんと崩壊を検出できるか確認する（測定器が甘くて異常を見逃さないかのテスト）

## 7. 指標そのものが分野の標準に照らして妥当か（何を測るかの検証）

測定器（項目6）が正しく動くかの前に、そもそも**「何を測るか＝指標の選択」がその研究課題にとって妥当か**を疑う。

- [ ] その指標は、この分野（発達ロボティクス・認知科学等）が実際に使っている評価基準か？ それとも汎用MLの直感で選んだ既定値か？
- [ ] 高コストな実験を回す前に、**ドメインの先行研究・文献が「これができた」をどう判定しているか**を確認したか？
- [ ] 一つの指標に絞らず、複数の側面（構造を掴めているか／較正まで合っているか）を分けて測っているか？
- 実例：**順モデルの評価に「持続予測に勝つ（絶対MSE）」を主指標にした**が、これは時系列予測の一般的なサニティチェックであって、**自己モデルの分野の標準ではなかった**。文献（発達ロボティクスの自己他者認知、乳児の随伴性検出）を当たると、実際の基準は「**自己（随伴）vs 他者（非随伴）の予測誤差の対比**」＝太郎の入替テスト(+22%)の方だった。しかも持続予測は「向き」と「量の較正」を混ぜて測っており、この課題には厳しすぎた。**本来の信号(+22%)は最初からデータにあったのに、非標準の厳しい bar に気を取られて過小評価していた**。→ 汎用MLの直感でbaselineを決める前に、ドメインの文献でどう評価するかを裏取りすべきだった（プロジェクトの原則「docsを鵜呑みにしない／思い込みで判断しない」を、自分の指標選びで破っていた）。

## 8. 壁の原因が「過去の機能」のことがある（コピー漏れ・変更の副作用）

新しい壁に当たったとき、「今の実装の新しいバグ」だけでなく、**過去に作った/変えた機能が原因**という可能性を必ず確認する。太郎は目標ごとにコードをコピーして育てているため、**欠落（コピー漏れ）と副作用**が起きやすい。

- [ ] その壁は、**別の目標(A/B)で実証済みの機構が、今の目標(C/D)にコピーされていない**ことが原因ではないか？（部分コピーの欠落）
- [ ] 過去の修正が、**別の場所に想定外の副作用**を与えていないか？
- [ ] チェック法：`comm`/`diff` で目標間の部品（`src/brain` 等）を突き合わせて欠けを探す／`git log`・`git blame` で該当箇所の変更履歴を辿る。
- 実例：**目標C2の自己モデルの頭打ちの真因は、目標Bにある海馬(睡眠リプレイ・記憶定着)が、Cへの部分コピーの際に来ていなかった**こと。「今のCの新バグ」を疑う前に消去法(lr/ノイズ/ホライズンを無罪)まで回して、ようやく「Bに在った定着機構がCに無い」と気づいた。**小脳(運動自動化)も同様に未コピー**＝次の壁の候補。→ 壁に当たったら早めに「**Bには在るのにCに無い部品はないか**」を diff で確認する。

## 9. 借りてきた環境の「課題」が、太郎の人生を勝手に打ち切っていないか

太郎は**環境の課題を無視して内発的動機で学ぶ**設計だが、**環境側の終了条件は生きたまま**のことがある。太郎が課題を無視していても、課題が太郎を打ち切る。

**実例（2026-07-15・D0で発覚。ほぼ全結果が汚染された）**：
`MIMoSelfBody-v0` を借りて自己接触を学習させたが、
- **`done_active=True`**（既定）＝環境の課題（目標部位を触れ）を達成するとエピソード終了。目標部位はランダムなので、**初期姿勢で既に触れている目標を引くと3 env.stepで終了**＝太郎は1回も判断（K=100）を完了できない。
- **`max_episode_steps=500`**（既定）＝太郎はK=100なので**5回判断しただけで打ち切り**。Cは6000（60回判断）だったので、D0だけ極端に短い人生だった。

**症状（＝この落とし穴のサイン）**：
- **`corr=0.998` / `persist=0.3%` という"良すぎる"数字**＝毎回同じ初期姿勢にリセットされ、それを予測していただけ（項6「都合が良すぎる結果は測り方を疑う」）。
- 録画が要求した長さの1/25しか撮れない、関節速度や触覚が**一定値**に張り付く。

**修正前後で数字が別物**（step600）：margin 33.7→**18.8**、corr 0.882→**0.253**、persist 22.3→**156.5**、飽和 28%→**4%**。＝汚染版の「良い数字」は細切れリセットの産物で、**「飽和が上がる」という"問題"自体が副産物だった**。

**チェック方法**：
- 借りた環境は必ず **`max_episode_steps` と終了条件（`done_active`/`is_success`）を確認**し、太郎の時間スケール（K）で割って**「1エピソード＝何回判断できるか」**を計算する。5回以下なら赤信号。
- **リセット回数をログに出す**（`n_reset`）。想定（n_train ÷ 1エピソードの判断回数）と合わなければ何かが切っている。
- 「何もしない行動」で1エピソード回して、**何ステップで終了するか**を実測する（0コスト診断＝項5）。

## 10. 数字だけ見て、行動を見ていないのではないか（指標は退化を高評価する）

**太郎の指標は「予測がうまいか」を測る。ところが予測を最もうまくする方法は「何も面白いことをしない」こと。** つまり **「モデルが良い」と「行動が良い」は逆相関しうる**＝指標は行動の退化を検出できないどころか**高く評価する**。これは運の悪さではなく、予測ベースのエージェントの**構造的な性質**。

**実例（2026-07-15・D0で3件連続）**：
| 数字（＝一見すると成功） | 実際に起きていたこと |
|---|---|
| margin+52.5 / corr0.757（**過去最高**） | 筋肉を全部最大活性にした**痙攣**（飽和100%・自己接触0%） |
| corr0.998 / persist0.3%（**ほぼ完璧**） | 環境が3stepでリセットし続け、**同じ初期姿勢**を予測していただけ（項9） |
| 自己接触65%（「創発した」と報告） | 腕を**可動限界に叩きつけていた**だけ |

3件とも**動画を見て初めて発覚**した。数字だけを見ていた側は全部「成功」と判断していた。

**対策（気をつける、では不十分。仕組みに埋める）**：
- **学習スクリプトの末尾に自動録画を組み込む**（`d0_selftouch.py` → `d0_record_trained.py`を自動実行）。**動画なしで数字だけ受け取ることを不可能にする**。
- **録画は等速で**。太郎の脳はK=100（0.5秒）に1回判断するので、判断ごとに1フレームだと**7.5倍速**になり、穏やかな動きも「暴れている」ように見える（誤診の元）。`realtime_every`で判断の"間"のコマも撮る。
- **「都合が良すぎる数字」は退化のサイン**として最優先で疑う（項6と対で使う）。
- 行動そのものの健全性を**指標化して併記**する（飽和率・関節速度・自己接触率など）。予測性能とは別軸で監視する。

## 11. 「合計」で測ると、次元数の多い方に薄められる（同じ日に2回踏んだ）

予測対象が複数の感覚から成るとき、**合計のMSEは次元数の多い感覚に支配される**。次元数の少ない感覚が完璧に学べていても、多い方が学べていなければ合計は「失敗」に見える。

**実例（2026-07-15・同じ罠を同日に2回）**：
| | 合計を見た私の報告 | 分けて測った実際 |
|---|---|---|
| **D0（座位）** | persist 86%＝「自己モデルが確立できていない」 | 固有感覚 persist 77.7% / margin **+39.8**＝確立している。触覚(89%の次元)が persist 96.3 で引きずっていただけ |
| **仰向け・触覚あり** | margin +15〜24＝「触覚ありは明確に劣化、範囲が重ならないのでノイズではない」 | 固有感覚 margin **+32〜46**（触覚なしの+40〜53とほぼ同等）。触覚(85%の次元)が persist 97〜104 で薄めていただけ |

2回とも**「3シードで範囲が重ならないから本物だ」と確信していた**。統計的に有意でも、**測っている量が間違っていれば意味がない**（項7と対で使う）。

**チェック方法**：
- 予測対象が複数の感覚を連結したものなら、**各感覚の次元数の比を必ず書き出す**。1つが70%を超えたら合計は使わない。
- **最初から分けて測るコードを書く**（`d_supine_split.py` / `d0_split_error.py`）。「あとで分ける」は忘れる。
- 学習済みモデルを保存しておけば、分けた測定は**学習し直さずに何度でもできる**（`C_SAVEMODEL`）。

## 12. 「条件の差」を語る前に「同じ条件のばらつき」を測る

n=1の比較は、ばらつきが差より大きければ**何も言っていないのと同じ**。

**実例（2026-07-15）**：動機の比較で6条件を n=1 ずつ回し、「学習進度が僅かに最良(+51.8 vs +50.8)」と報告した。ところが**同一条件・同一シード**の2ランが **46.2 vs 57.0（10.8ポイント差）**。**ばらつきが条件差(1.5)の7倍**＝その日の比較は全て無効だった。
ズレた原因は項3（朝のランは`C_INVTRAJ=1`で各checkpointに診断を挟み、その診断が乱数を消費して学習の乱数列をずらした）。

**チェック方法**：
- **同じ条件を最低3回**回し、その幅を出してから条件間を比べる。幅より小さい差は語らない。
- 診断・プローブ類は**比較する全条件で揃える**か、全部切る。片方だけ診断ONは条件差ではなく乱数差を測っている。
- 並列数が足りないなら**メモリを削る**（2026-07-15：MIMoのテクスチャ976MBを落として1本2.64GB→0.28GB＝6本→22本並列。`D/scripts/mimo_lean.py`）。シード数は買える。

## 使い方
- 測定結果を根拠に結論を出す前に、1〜12を一通り確認する。
- 特に「予想と違う／都合が良すぎる／実行ごとにブレる」結果が出たら、まず測り方を疑う。
- 新しい発見をしたら、それが「本物」か「測定の副産物」かを、上記で切り分けてから記録する。
