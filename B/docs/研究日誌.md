Bridges（1932）の感情分化説を採用。生まれた瞬間は「つらさ」の1次元だけ。「うれしい」や「こわい」は後で経験から枝分かれする。だから太郎にも最初は感情の枠すら作らない。

* カーブは数式で決め打ちしない。メカニズムを作って結果を出す
* 臓器を増やすのではなく、臓器間の接続を足す
* 脳に渡す信号が変わる部分だけ精密にする
* 最初は粗く、合わなくなったら1段階ずつ上げる

#### オウム返しはBでも出る（ただし10回連続の判定基準を満たすのが難しい）

### 考察

#### 「胃空 = 空腹MAX」は設計ミスだった

B-2では `hunger = 1 - stomach.contents / capacity` という直接対応だった。人間の空腹感は血糖値が下がるサイン。血管（バッファ層）を挟むことで：

* 食後は血糖値が維持されるため空腹感が低い（2〜3時間）
* 胃が空になってからではなく、血糖値が落ちてきたときに空腹になる

#### 定期授乳の廃止で因果関係が成立した

B-2の定期授乳は「腹が減った → 泣く → 授乳される」という因果関係を壊していた。B-3からは泣き→授乳のサイクルのみになり、「まんま」という音に意味が付く前提が整った。

#### 自発喃語が「発達段階の橋渡し」になる

B-2まで太郎は親に話しかけられた34回しか発声しなかった。B-3では喃語240回が加わり、発達段階（喃語 → オウム返し → 初語）に必要な練習量が確保された。

### 設計メモ

#### 声道（VocalTract）の設計上の課題

`update\_stage()` は `taro/body/vocal\_tract.py` に定義されており、Taroの体オブジェクト上のメソッドとして存在している。しかし「声道を成熟させる」という能動的なメソッドをTaroが持つのは人間模倣として不自然（人間は意識的に声道を成熟させない）。

将来的には「声道は生まれついての制約として最初から正しい段階に入っている」設計が理想——つまり声道オブジェクトは `get\_available\_phonemes(sim\_seconds)` のように時刻を受け取って現在使える音を返すだけにして、`update\_stage()` のような「状態を書き換えるメソッド」は持たない。今は許容範囲として残す。

### 考察

#### 1シミュレーション秒 ≠ 1現実秒だった

B-4まで `stage1\_time: 300` という設定は根拠のない仮置きだった。人間の規準喃語（ままま・ばばば）は生後6〜10ヶ月から始まる。1シミュレーション秒を1現実秒として設計すると、stage1 は180日×86400秒 = 15,552,000秒が正しい。

この修正により：

* 7日間シミュレーションでは母音のみ（クーイング期）が正しく再現される
* 初語（まんま）検証には最低9〜14ヶ月相当のシミュレーションが必要になる

4. **原因調査：なぜ hunger→まんまの学習が起きないのか**

   コードを追うと、`brain.generate()` が `body\_state` を受け取っていなかった。

   * 知覚学習（`learn\_perception`）には body\_state を渡していた ✓
   * 生成（`generate()`）の内部ループでは `forward\_hidden(token, hidden)` を `body\_state=None` で呼んでいた ✗
   * 発声の各ステップで Insula 出力が **常にゼロ** → GRU は hunger を参照できない

   * `core\_b.py`：`step()` と `self\_babble()` の `brain.generate()` 呼び出しに `body\_state=body\_state` を追加

   #### 知覚と生成で body\_state が非対称だった

   学習（知覚）時は hunger が GRU に届いていた。しかし生成時は届いていなかった。これにより GRU は「空腹+まんまを聞く → 次はんが来る」を学習しても、「空腹 → まんまを出す」を生成時に活用できなかった。B-9 でこの非対称を解消した。

   乳児が発話を止めるメカニズムは（1）呼吸制御、（2）ターンテイキングの社会的同期、（3）音節単位の運動プログラム終了 が複合している。どれも現在の太郎には未実装の高次機能。肺活量（morae 数）で長さを自然に制限する現在の設計が当面は最も人間模倣に近い。

    「ま」「ん」の音は使えるようになり類似度平均は0.32まで上昇したのに、6000件以上の機会があっても
    一度も閾値0.4を超えなかった。**音素は使えても、正しい並び（ま→ん→ま）を出す学習が
    全く起きていない**ことが分かった。

18. **B2-2：REINFORCEのクレジット割り当てを改善**

    ユーザーから「閾値はできるだけ使いたくない（人間にはないから）、根本的に改善したい」との
    方針が示された。閾値の調整ではなく、上記の「並びを学習できない」原因そのもの
    （REINFORCEが発話全体に単一のδしか与えず、文字ごとの良し悪しを区別できない）に対応。

    - `imitation.py`に`compute_alignment_credit()`を新設。模倣報酬の計算に既に使っている
      重み付き編集距離のDPテーブルをバックトレースし、生成した文字ごとに目標語のどの位置と
      どれだけ一致していたかを求める
    - `basal_ganglia.py`の`learn_action()`に`credits`引数を追加。文字ごとの一致度でδを
      重み付けし、良かった文字はより強く強化、目標語にない余分な文字は抑制する
    - `step()`と`respond_to_babble()`の両方に組み込んだ
    - 新しい決め打ちパラメータ（閾値等）は一切追加せず、既存の音声距離計算を再利用しただけ

    単体テストで動作確認：target="まんま", generated="まんみ" → credit=[1.0, 1.0, 0.75]
    （最初2文字は完全一致、3文字目は部分点）。意図通り文字ごとの一致度が反映されている。
    185日統合テストも完走。

### 考察

#### 「音素が使えること」と「正しく並べられること」は別の学習

声道成熟で「ま」「ん」が物理的に出せるようになっても、それだけでは「ま→ん→ま」という
特定の並びを高い確率で選ぶようにはならない。当たり前のようだが、今回の週次分析で
初めて定量的に確認できた。REINFORCEが発話全体に単一の報酬しか与えない設計では、
「並び」を学習する力がそもそも弱かったことが根本原因だった。

#### 決め打ちのパラメータを避けたいという方針が、より良い設計を導いた

閾値を下げるという対症療法ではなく「なぜ閾値を超えられないのか」を掘り下げたことで、
より本質的な問題（クレジット割り当て）にたどり着けた。「人間にはない仕組みは極力避ける」
という一貫した方針が、結果的に安易な数値調整より筋の良い修正につながった。

#### 「文字ごとに一致度を計算」は人間模倣ではなかった（ユーザー指摘で訂正）

`compute_alignment_credit()`の実装当初、これを【人間模倣】としてDIVAモデルに紐づけて
説明していたが、ユーザーから「実際の人間は文字ごとに一致度を計算していないのでは」と
指摘を受け、調べ直したところ誤りだったと判明した。乳児の語彙表現は単語・音節単位の
「まとまり」として持たれ、言葉を1音ずつ分解して比較する音素意識（phonemic awareness）は
発達的に最も遅く獲得されるスキルで、多くは4〜6歳の読み書き学習と結びついて初めて育つ。
生後1年の太郎がこの計算を実際にやっているとは考えにくい。

正しい位置づけは：「まんま」という目標語自体はまとまりとして扱ってよい（ここはDIVAの
考え方とも矛盾しない）が、そこに至る運動修正が発話全体を待たず継続的に起きるという
"原則"はDIVAに沿っているものの、それを実現する具体的な手段（編集距離のバックトレースで
文字ごとに信頼度を出す）は人間の脳がやっていない計算方法。⚠️逸脱として正直にラベルを
貼り直した（`imitation.py`, `basal_ganglia.py`のdocstring・版管理.mdを修正）。
backpropと同様、「既存AI研究の手法だが人間模倣ではない」ものとして扱う。

    - 停止確率を`息切れ圧力 × (1 - 興味度)`に変更。「疲れたし退屈」なら
      すぐ止まる、「疲れたけどまだ面白い」なら頑張って続ける、という
      現実的な組み合わせになった
    - 既存のサンプリング分布（`_choose_param`が使っているのと同じ確率分布）を

    `log_prob`を`_apply_ne_noise()`の後に、実際に発声される値から
    `_log_prob_of()`で計算し直すよう修正。既存の関数（発話計画の
    「口の動きが既知」の場合に既に使われていたもの）を再利用するだけで、
    新しい決め打ちパラメータは追加していない。

    単体テスト・185日統合テストで動作確認、エラーなし。

### 考察

#### 「もっともらしい仮説」は必ず裏取りしてから直す

B2-5の「息切れ」修正と同じ構造で、今回も「NEノイズがズレていそう」という
仮説をユーザーに問い直されて初めてコードを直接検証した。もし最初から
自分でコードを読んで確証を取る習慣がついていれば、後から指摘されずに
済んだはず。「もっともらしい」と「実際にそうなっている」は別物という
教訓が、このプロジェクトで繰り返し確認されている。

#### 診断実験という手法自体の有用性

自然な相関だけでは「効果がない」のか「効果はあるが測定できていない」のか
区別できない、という問題は、方法論エージェントが以前指摘した検定力不足の
話と同根。人工的に条件を固定して比較する診断実験は、こうした場合に
効果の有無を白黒つける手段として有効だと今回確認できた。今後も
「相関が弱くて解釈に困る」場面ではこの手法を検討する価値がある。

30. **B2-6の12ヶ月シミュレーションを再実行（単一試行で確認）**

31. **「単一試行問題」への対応：`run_12months.py`を複数試行対応に改修**

    94エージェント議論で指摘されていた「乱数シード1回だけの結果で
    版の効果を判断している」という問題に対応。コマンドライン引数で
    trial IDを渡すとログファイル名・run_nameにサフィックスが付く
    ようにした。各Pythonプロセスは起動時にOSから独立に乱数シードを
    もらうため、同じスクリプトを複数プロセス同時起動するだけで
    素朴な複数試行になる（詳細な乱数管理は不要）。

    1試行あたりスレッド数2に制限しているため、2試行を同時実行しても
    合計4スレッド程度でCPU負荷は軽い。

32. **trial1・trial2を同時実行し、単一試行の結果と比較**

    | 指標 | 元(single) | trial1 | trial2 |
    |---|---|---|---|
    | まんまを含む喃語 | 633件(0.617%) | 1045件(1.013%) | 608件(0.593%) |
    | hunger-類似度 相関r | +0.0338 | +0.0601 | +0.0726 |
    | hunger>0.5時 類似度 | 0.3352 | 0.3398 | 0.3343 |
    | hunger<0.3時 類似度 | 0.3211 | 0.3168 | 0.3063 |
    | 喃語平均長 | 3.21 | 3.21 | 3.24 |

    3試行とも相関の符号は一貫して正（+0.03〜+0.07）、hunger高時の
    類似度がhunger低時を常に上回り、「まんま」を含む喃語が0件に
    なることは一度もなかった。B2-6の効果は乱数の偶然による単発の
    観測ではなく、再現性のある傾向だと確認できた。

### 考察

#### 3試行は「統計的証明」ではなく「単発でないことの最低限の確認」

3試行だけでは分散や信頼区間を厳密に論じることはできない。しかし
94エージェント議論が問題視していたのは「n=1のまま版ごとの優劣を
語っている」点であり、n=3で符号が一貫することはその批判への
直接的な反証になる。相関の大きさ自体が0.034〜0.073と2倍以上
ブレている点は、効果がまだ小さく測定誤差の影響を受けやすい
ことを示しており、今後さらに試行数を増やす／効果量そのものを
大きくする余地があることも同時に示している。
